Algorithm Comparison

This demo shows two comparison algorithms side-by-side:

❌ VULNERABLE (Timing Leaks)

Uses if/else branches that execute at different speeds. Timing reveals the result.

Vulnerable Code (LEAKS TIMING)
bool vulnerable_compare(string a, string b) { if (a == b) { return true; // Fast: ~10 cycles } else { return false; // Slow: ~100 cycles } } // PROBLEM: Attacker measures timing // Fast = match, Slow = no match
Secure Code (CONSTANT-TIME)
uint32_t diff = secretA ^ secretB; // XOR: 0 if equal, non-zero if different uint32_t match = ((diff | -diff) >> 31) ^ 1; // Normalize result to 1-bit // Always same instructions, always same time // ADVANTAGE: Timing is identical! // No information leaked!

📚 How This Works

1. Local Hashing: Your secrets are hashed locally in your browser using SHA-256. Plaintext never leaves your device.

2. Constant-Time Comparison: The encrypted hashes are compared using branchless bitwise operations. Every comparison takes exactly 50ms, whether secrets match or not.

3. Host Blindness: The server receives only encrypted payloads and a 1-bit result. It cannot infer anything from timing because execution time is constant.

4. Single-Bit Output: Only MATCH or NO_MATCH is returned—no additional information leaks.
